## D. 执行步骤（对齐 pig-scaffold skill 9 步）

### 步骤 1 · 克隆 & 改名

- `git clone` 后端脚手架到 `devolop-master/`
- `git clone` PC 前端脚手架到 `exhibit-flow-vue/`，修改 `package.json.name` / `index.html` 的 title/keywords/description 为"展厅访问申请与留存系统"

### 步骤 2 · 创建数据库

- 本地默认 MySQL 服务（127.0.0.1:3306）
- `CREATE DATABASE exhibit_flow DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;`
- **先导入 `devolop-master/db/` 下的系统初始化 SQL**（用户/角色/菜单/字典/OAuth2 等 `sys_*` 表），完成 Pig 基础环境
- 再执行步骤 3 的业务表 DDL

### 步骤 3 · 建表（依据功能清单）

按前面方案确定的 8 张业务表建表，统一 `eh_` 前缀，强制公共字段（`id / create_by / create_time / update_by / update_time / del_flag`）：

| 表 | 用途 |

|---|---|

| `eh_apply` | 申请主表 |

| `eh_apply_visitor` | 参观客户明细（1 对多） |

| `eh_approval_node` | 审批节点流水 |

| `eh_visit_record` | 参观留存 |

| `eh_visit_photo` | 现场照片 |

| `eh_opportunity` | 商机跟进 |

| `eh_change_log` | 改期/取消流水 |

| `sys_dict` | 复用 Pig 自带字典（省/市/行业/战客级别录入） |

执行建表使用 mysql skill，加 `--default-character-set=utf8mb4`。

### 步骤 4 · 后端四层代码生成

在 `pig-boot/src/main/java/com/pig4cloud/pig/eh/` 下生成：

- `entity/` — 7 个 Entity（不含 sys_dict）
- `mapper/` + `resources/mapper/eh/*Mapper.xml`
- `service/` + `service/impl/`
- `controller/` — 每表标准 5 接口（page / details / save / update / remove）
- 权限注解前缀：`eh_apply_*`、`eh_visit_*` 等
- 用 Python 脚本批量生成（避开 zsh heredoc 中文变量问题）

### 步骤 5 · 配置修改

- `application-dev.yml`：数据源 url 指向 `exhibit_flow`，本地 password 去掉 `ENC()`，`hikari.pool-name` 改 `ExhibitFlowHikari`
- 确认 `@MapperScan("com.pig4cloud.pig.**.mapper")` 已覆盖

### 步骤 6 · 菜单权限

- menu_id 从 `11000` 起编排
- 一级菜单：展厅申请管理（11000）、参观留存（12000）、商机跟进（13000）、统计报表（14000）、基础数据（15000）
- 每个二级菜单挂载 view/add/edit/del/approve/export 按钮权限
- 使用 mysql skill 插入，必须 `--default-character-set=utf8mb4`

### 步骤 7 · PC 端代码生成

- `src/api/eh/*.ts` — 每业务模块一个 API 文件，URL 以 `/admin/eh/` 前缀
- `src/views/eh/{模块}/index.vue` — 套用脚手架标准模板（el-card 查询 + el-table + el-dialog）
- 路由自动由菜单驱动（Pig 特性），无需手写

### 步骤 8 · 移动 H5 端（仅架子，**本次重点**）

**技术栈：** Vite 5 + Vue 3 + TypeScript + Vant 4 + Vue Router 4 + Pinia + axios

**初始化方式：** `npm create vite@latest exhibit-flow-h5 -- --template vue-ts`，然后按需加装

**目录结构（空壳）：**

**不做的事：**

- 不创建申请表单、申请列表、留存录入、照片上传、商机补录等业务页面
- 不写业务接口函数（仅留 `request.ts` 基础设施）
- 不接入 UI 设计稿（用户尚未设计）

**可跑通的验收标准：**

- `npm run dev` 能起,访问 `http://localhost:5173`
- 登录页能提交账号密码,成功后跳转首页
- 首页显示占位文案,TabBar 显示但点击无跳转(路由未配置业务页)
- 401 能正确跳回登录

### 步骤 9 · 编译 & 启动

- 后端：`mvn compile -pl pig-boot -am -DskipTests -q` → `mvn spring-boot:run -pl pig-boot`
- PC 端：`cd exhibit-flow-vue && npm install && npm run dev`
- H5 端：`cd exhibit-flow-h5 && npm install && npm run dev`
- 三端同时跑通即为初始化完成

## E. 关键决策点

| 决策 | 选择 | 原因 |

|---|---|---|

| H5 端是否纳入 Pig monorepo | **独立仓库/目录** | Pig 脚手架自身不含 H5 模板；保持 H5 轻量,独立迭代 |

| H5 端登录是否复用 Pig OAuth2 | **复用** | 共用账号体系,避免再造;用 `/oauth2/token` 密码模式拿 access_token |

| H5 端路由模式 | **Hash 模式** | 部署最简单,静态文件直接丢 nginx 或 Pig 的静态目录 |

| H5 端是否先推 gitee | **否,本地初始化即可** | 用户尚未准备远程仓库;后续手动 `git init` 推送 |

| Entity 生成方式 | **Python 脚本** | pig-scaffold 明确要求,避免 zsh 中文变量问题 |

## F. 已确认决策

- **MySQL**：本地默认服务（127.0.0.1:3306），新建 `exhibit_flow` 库，先跑 clone 下来的系统初始化 SQL（sys_* 表），再建业务表
- **H5 联调**：开发期走 Vite `proxy` 将 `/admin` 转到后端本地 9999，免跨域
- **H5 微信 JS-SDK**：本次仅搭架子，暂不集成（后续拍照/定位需要再加）
